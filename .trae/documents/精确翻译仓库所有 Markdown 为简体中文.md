## 目标
- 将仓库内全部 `*.md` 文件内容精确翻译为简体中文，保持原有 Markdown 结构与格式不变，并直接覆盖原文件内容。
- 不使用任何外部工具或服务，仅通过模型能力完成逐段翻译与术语统一。

## 范围与文件
- 处理范围：仓库根目录下匹配 `**/*.md` 的全部文件（含 `README.md` 及各日期命名的文档）。
- 文件名、目录结构不变；仅替换每个 `.md` 文件的文本内容。

## 翻译原则
- 保留不翻译内容：
  - 代码块（```fence```）、内联代码（`...`）、命令、配置、示例数据、JSON/YAML、路径和环境变量。
  - URL、文件路径、版本号、日期、产品/组织专有名词（如 OpenAI、Anthropic、GitHub 等）。
- 翻译内容：
  - 正文段落、标题、说明文字、列表项、表格单元中的自然语言文本。
  - 链接的可见文本翻译，但不改动链接目标（href）。
  - 图片的替代文本（alt）翻译，但不改动图片路径。
- Markdown 规则：
  - 保持原有层级（#、##、###）、列表缩进、引用块、表格分隔线与对齐、粗斜体标记等不变。
  - 保持代码块语言标签不变（如 ```js、```python）。
- 术语统一（示例）：
  - system prompt → 系统提示词
  - assistant → 助手
  - agent → 智能体
  - tool → 工具
  - codebase → 代码库
  - model → 模型
  - workspace → 工作区
  - patch → 补丁

## 实施步骤
1. 枚举并确认所有 `.md` 文件列表（已完成初步扫描，后续以 `**/*.md` 作为准则）。
2. 逐文件读取：按段落、标题、列表等块级结构进行精确翻译，严格遵守“保留不翻译内容”规则。
3. 术语一致性：对易混术语执行二次校对，确保全仓统一表达。
4. 写回覆盖：用中文译文替换原文件内容，文件名与路径不变。
5. 质量检查：
   - 逐文件自检：格式未破坏、代码块未改动、链接指向未变。
   - 全仓抽检：随机抽查若干文件，核对术语和标题层级。

## 风险与处理
- 混排文本风险：代码与说明混排时，采用“块内逐句判断”，只翻译说明，不触碰代码与命令。
- 锚点变化风险：若文内存在对标题的锚链接（`#heading`），翻译可能改变锚文本；处理方式为保留链接目标不变，仅翻译可见文本，避免破坏现有链接。
- 专有名词误译：对产品名与组织名一律保留原文。

- 验证要点：随机抽查 3–5 个文件，确认格式完整、链接与代码块无误、术语统一。
